## Purpose
This directory implements the Vox Debug Workbench, a diagnostic tool for monitoring and inspecting processing requests. It provides a centralized system for capturing request statuses, raw transcripts, rewritten outputs, and execution logs. The workbench includes a graphical interface for real-time visualization and interaction with request history.

## Key Roles
- **DebugWorkbenchSink**: A sendable interface that routes request updates, logs, and transcript data to a storage backend; it provides a live implementation that bridges to the main actor.
- **DebugWorkbenchStore**: An observable, main-actor-isolated class that manages the state of request records, including status tracking and output pane management.
- **DebugWorkbenchView**: A SwiftUI view hierarchy that renders request cards, status badges, and navigable output panes for transcripts and logs.
- **DebugWorkbenchWindowController**: An AppKit window controller that hosts the debug interface in a floating, resizable window.
- **DebugWorkbenchRuntime**: A utility for determining if the workbench is active based on environment variables and build configurations.

## Dependencies and Caveats
- Depends on `VoxCore` for `ProcessingLevel` and underlying data types.
- Uses `AppKit` for window management and `NSPasteboard` for clipboard operations.
- The workbench is only enabled in `#if DEBUG` builds when the `VOX_DEBUG_WORKBENCH` environment variable is set to "1", "true", "yes", or "on".
- The store enforces a limit of 200 requests and 150 log lines per request.
- Updates from `DebugWorkbenchSink` to the store are performed asynchronously via `Task { @MainActor in ... }`.
- The debug window is configured as a floating level window that can join all spaces.
- UI components utilize `LazyVStack` for rendering the request list and include built-in text selection and copy-to-clipboard functionality.
