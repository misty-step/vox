## Purpose
This directory contains unit tests for the VoxDiagnostics module, focusing on the validation of diagnostic data collection, error mapping, and persistence. It ensures that system errors are correctly translated into standardized diagnostic codes, events are serialized accurately to JSONL format, and log files are managed through rotation and export processes.

## Key Roles
- **DiagnosticsErrorCodeTests.swift**: Validates the mapping of domain-specific errors—including STT, Streaming STT, Rewrite, Vox, and Audio Conversion errors—into standardized string codes and verifies the serialization of diagnostic values and events.
- **DiagnosticsStoreTests.swift**: Tests the recording of events to disk, log file rotation and pruning logic based on file size and count limits, and the generation of diagnostic ZIP exports containing system context and logs.
- **ProductInfoTests.swift**: Verifies the resolution of application metadata, such as version and build numbers, from bundle info dictionaries or environment variable overrides.

## Dependencies and Caveats
- Depends on the `Testing` framework for suite and test definitions.
- Utilizes `Foundation` for file management, JSON serialization, and process execution.
- Requires `VoxCore` for domain-specific error types used in mapping tests.
- Uses the system utility `/usr/bin/ditto` via `Process` to validate the integrity of exported ZIP archives in tests.
- Log rotation tests are grounded in the interaction between `maxFileBytes` and `maxRotatedFiles` parameters.
- Product metadata resolution includes hardcoded fallbacks for development environments and specific attribution strings.
- Diagnostic values support specific data types including strings, integers, doubles, and booleans, with verified round-trip JSON codability.
