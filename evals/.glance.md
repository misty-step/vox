## Purpose
This directory provides a framework for evaluating AI model performance and safety when processing automatic speech recognition (ASR) transcripts. It integrates prompt templates, evaluation datasets, and configuration files to test transcription cleaning and polishing capabilities across various language models. The system specifically validates instruction-following, prompt injection resistance, and the prevention of hallucinations or unauthorized command execution. Evaluation is orchestrated through automated scripts and multi-model "bakeoffs" using the `promptfoo` framework.

## Key Roles
- **bakeoff-config.yaml**: Configures multi-model safety and quality evaluations for transcription cleaning using the `smoke.yaml` dataset.
- **polish-bakeoff.yaml**: Defines a multi-tier model evaluation (Baseline, Value, and Premium providers) for structural polishing using the `polish.yaml` dataset.
- **polish-smoke.yaml**: Provides a configuration for rapid sanity checks of the polishing prompt to ensure no hallucinations occur.
- **promptfooconfig.yaml**: Sets the primary configuration for rewrite quality evaluations using a specific `llm-rubric`.
- **datasets/**: Contains YAML-based test cases covering prompt injection techniques, transcript polishing scenarios, and general smoke tests.
- **prompts/**: Stores JSON templates defining "transcription editor" and "elite editor" roles for minimal cleaning or structural rewriting.
- **scripts/**: Includes shell scripts and Node.js utilities for executing evaluations, parsing results, and generating Markdown reports.

## Dependencies and Caveats
- Evaluations rely on the `promptfoo` framework and `npx` for core execution and orchestration.
- Quality judgment is managed via `llm-rubric` assertions with defined scoring thresholds ranging from 0.7 to 0.75.
- Model providers are primarily accessed through OpenRouter, including variants of Gemini, Qwen, GPT, Claude, Llama, and Mistral.
- Safety rules strictly forbid models from fulfilling commands found within transcripts, such as requests to write poems or lists.
- Evaluation configurations utilize `maxConcurrency` limits (2 or 3) and explicitly disable caching.
- Reporting utilities depend on specific JSON structures provided by `promptfoo` results, including `gradingResult` and `componentResults`.
- Deterministic assertions, such as `contains`, `not-contains`, and Levenshtein distance, supplement LLM-based judging.
- Node.js scripts require `fs` and `process` modules for file I/O and handling evaluation result streams.
